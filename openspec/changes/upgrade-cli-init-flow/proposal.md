## Why

Currently, the `init` command in `only-one` always runs a full sequence of steps (tools, packages, skills, configs) or combo selections. Users want more granular control to initialize specific resources individually, reuse initialization logic programmatically, and automatically manage their `.gitignore` file for configured tools/skills to prevent accidental commits of local files.

## What Changes

- Refactor `init` command logic to support nested subcommands: `init combo`, `init skill`, `init package`, and `init configs`.
- The main `init` command will continue to run the current interactive wizard or combo flow, reusing the code of the subcommands.
- Support options/arguments on subcommands to allow direct installation (e.g., `only-one init package typescript,prettier`) or interactive multi-select when arguments are omitted.
- Add an automatic `.gitignore` update mechanism that appends configuration directories/files generated by agent tools or skills to the project's `.gitignore` file.
- Add a new `--no-ignore` option globally to the `init` command and subcommands to opt out of the automatic `.gitignore` updates.

## Capabilities

### New Capabilities
- `init-subcommands`: Support running specific initialization scopes (`combo`, `skill`, `package`, `configs`) with dedicated subcommands and options.

### Modified Capabilities
- `init-interactive-flow`: Update the interactive wizard flow to reuse subcommand logic, support the `--no-ignore` flag, and automatically update `.gitignore`.
- `init-combo-flow`: Support the `--no-ignore` flag to disable updating `.gitignore` when initializing via combo.

## Impact

- CLI entrypoint (`src/cli/index.ts`) will register subcommands under `init`.
- `src/commands/init/command.ts` will define subcommands and their option parsing.
- `src/core/init/init-command.ts` will be refactored to modularize individual step execution so that subcommands and the parent `init` flow share code.
- A new utility for `.gitignore` updates will be added.
